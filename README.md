# YieldGuard

A two-stage ad ranking system: two-tower retrieval, a GBDT reranker, isotonic
calibration, and a toy second-price auction — with an offline evaluation
harness that reproduces every number below from one command. The name is
the point: an uncalibrated model quietly erodes yield (see the auction
simulation below), and this pipeline exists to catch that before it ships.

```
python -m src.eval.run --config configs/gbdt.yaml
```

`results/metrics.json` is regenerated by that command and is what every
number here is read from. `docs/design.md` records the reasoning behind
each design decision, including two I got wrong on the first pass and had
to fix (see below) — it's written to double as interview prep.

## Why this exists

Most portfolio ranking projects are a notebook that trains a model and
reports AUC. This one is closer to what an actual ads-ranking system has to
get right: a candidate set too large to score exhaustively, a probability
that has to be genuinely correct (not just well-ordered) because money
changes hands based on it, and an offline metric suite that a CI job
re-verifies on every push.

## Architecture

**Offline (nightly training):** raw click logs → feature pipeline (temporal
split, leakage-safe user/category history) → two-tower model (retrieval
embeddings) + GBDT ranker → isotonic calibrator → FAISS index + versioned
model artifacts.

**Online (serving, FastAPI):** retrieve (ANN, top ~500) → rerank (GBDT on
just those candidates) → calibrate (isotonic map) → auction (rank by
calibrated pCTR × bid). Every stage is timed and exported as a Prometheus
histogram.

## Results

All numbers below are from `results/metrics.json`, seed 42, on synthetic
data (see "Why synthetic data" below) — 480K train / 60K val / 60K test rows.

### Ranking quality

| model | AUC | log loss |
|---|---|---|
| logistic regression (hashed features) | 0.716 | 0.177 |
| GBDT (LightGBM) | 0.715 | 0.274 |

AUC is essentially tied — on this data the log-odds structure is fairly
additive, so a linear model captures most of the ranking signal. The GBDT's
actual advantage shows up on personalized ranking, not raw AUC:

| | NDCG@10 |
|---|---|
| GBDT reranker | **0.202** |
| popularity baseline | 0.166 |

(1,374 users evaluated, 100 sampled negative candidates per user — a true
click plus 100 ads the user didn't interact with, scored together; see
`sampled_ranking_eval` in `src/eval/run.py`.) The GBDT beats popularity
because it has a feature popularity structurally cannot use: this user's
historical CTR within *this ad's category*. Getting this comparison honest
took two fixes — see "What I got wrong" below.

### Calibration: the centerpiece result

The GBDT is trained on negative-downsampled data (keep all clicks, 20% of
non-clicks) — standard practice for large-scale CTR training, and it has a
well-known side effect: the raw score no longer reflects the true click
probability.

| | Expected Calibration Error |
|---|---|
| raw GBDT score | 0.147 |
| after isotonic calibration | 0.0018 |

That's not just a metrics-page number — it has a direct dollar cost. A toy
second-price auction (`src/eval/auction_sim.py`) runs two otherwise-identical
bidders through the same impressions with the same bids, one pricing off the
raw score and one off the calibrated score, against a fixed reserve floor:

| | impressions won | spend | true value generated | ROI |
|---|---|---|---|---|
| uncalibrated | 56,068 | $31,783 | $7,893 | **-75.2%** |
| calibrated | 19,609 | $5,504 | $5,376 | **-2.3%** |

The uncalibrated bidder wins far more auctions than it should (its inflated
score clears the reserve too easily) and pays for traffic that doesn't
convert at the rate it thinks it does. Calibration doesn't touch ranking at
all — it only rescales the score — and recovers ROI from catastrophic to
near break-even.

### Retrieval

FAISS retrieval (top 200 of 2,000 ads) achieves **10.6% recall@200** against
the ads users actually clicked — i.e. of the ads a user would have clicked,
about 1 in 10 survive the retrieval cut before the reranker even sees them.
This is the honest number for a two-tower model trained for a handful of
epochs on a modest embedding size; it's the clearest opportunity for
improvement in this repo (more epochs, hard-negative mining, or a larger
embedding dimension would all likely help — none of that is done here, to
keep the reported number real rather than tuned).

## What I got wrong on the first pass (and how it was caught)

Two of the numbers above only exist because an earlier version of this
project produced a *worse* result and I chased down why, rather than
reporting it:

- **The GBDT initially lost to the popularity baseline on NDCG.** The
  category-personalization feature existed but was too sparse to learn from
  (1.5 impressions per user per category on average) — so the model had
  nothing real to exploit and popularity legitimately won. Fixed by reducing
  the synthetic user count so the same data volume gives each user enough
  repeat signal per category (see `docs/design.md`).
- **The GBDT was already almost perfectly calibrated**, which meant the
  calibration stage had nothing to correct and the whole "calibration
  matters" section would have been an assertion with no evidence behind it.
  Fixed by training on realistically negative-downsampled data instead of
  the full class balance, which introduces the exact kind of miscalibration
  a real production CTR pipeline has to correct for.

Both are documented in full in `docs/design.md`, including the mechanism
and the numbers before/after the fix.

## Repo layout

```
configs/          experiment configs (yaml)
data/generate.py  synthetic click-log generator (see below)
src/features/     feature pipeline, temporal split, leakage guards
src/models/       LR baseline, GBDT ranker, isotonic calibrator, negative downsampling
src/retrieval/    two-tower model, FAISS/ScaNN index
src/eval/         metrics, auction simulation, the one-command eval runner
src/serve/        FastAPI app, artifact store
src/train.py      trains and saves every serving artifact to models/{run_id}/
tests/            incl. test_no_leakage.py -- the one test that matters most
docs/design.md    decisions, trade-offs, and what I got wrong
```

## Why synthetic data instead of Avazu/Criteo

Avazu (40M rows) and Criteo need a Kaggle account and a multi-GB download,
neither of which reproduces in CI. `data/generate.py` produces a smaller
dataset with the same structural properties that make CTR modeling
non-trivial: extreme class imbalance (~5% CTR), high-cardinality IDs that
must be hashed, a real time axis, user-level heterogeneity, and a genuine
per-user × category preference signal. Every number in this README comes
from this generator with a fixed seed (`--seed 42`), so the whole pipeline
reproduces on a laptop with no external downloads or credentials. See
`docs/design.md` for the full rationale and every parameter choice.

## Running it

```bash
pip install -r requirements.txt
pytest                                          # incl. the leakage guard
python -m src.eval.run --config configs/gbdt.yaml   # reproduces every number above
python -m src.train --config configs/gbdt.yaml --run-id latest  # trains serving artifacts
YIELDGUARD_RUN_DIR=models/latest uvicorn src.serve.app:app --reload  # POST /rank
```

CI (`.github/workflows/ci.yml`) runs the test suite plus the eval pipeline
on a smaller sample on every push.

## What isn't built

No production feature store (training and serving share code paths here,
which a real system wouldn't), no online A/B testing harness, no position-
bias correction, and no cold-start model beyond a Laplace-smoothed prior.
Every one of these is flagged in `docs/design.md` along with what would
actually be needed at real scale.
